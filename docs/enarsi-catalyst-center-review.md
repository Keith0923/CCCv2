# CCCv2 レビュー（本番GUI再現度フォーカス）

## 1. 重点観点

本レビューは以下3点に限定して評価する。
- 表示粒度（KPI・詳細・履歴の見せ方）
- 連携文脈（画面間の遷移理由と状態引き継ぎ）
- 操作統一（同種操作のUI配置・命名・挙動）

## 2. 結論（先に要点）

- **現状判定: 70/100（導線は良いが、画面間の一貫性で減点）**
- 改善優先は「連携文脈」→「操作統一」→「表示粒度」の順。

## 3. 詳細レビュー

### 3.1 表示粒度

#### 良い点
- Discoveryは結果要約（success/partial/failed）を即確認できる。
- Inventoryは一覧＋右railで調査/変更の同時把握が可能。

#### 改善点
- 右railの情報密度が画面ごとに揺れており、詳細の深さが不統一。
- 同じ「状態確認」でも、表・要約・タグの出し分け規則が明文化されていない。

### 3.2 連携文脈

#### 良い点
- AssuranceからDevice 360/Troubleshooting/Path Traceへ到達できる。
- Inventory→Device 360、Topology→Inventory/Device 360の往復が成立。

#### 改善点
- 遷移先はあるが「なぜこの遷移が次に適切か」の表示が弱い。
- site/device/issueの受け渡し表示が画面で揺れ、調査文脈が切れやすい。

### 3.3 操作統一

#### 良い点
- Filter/Action/Table/Railという基本レイアウトは多くの画面で共通。

#### 改善点
- 同じ操作意図（Focus/Open/Select）が画面ごとに別語・別配置で現れる。
- 主要アクションの優先順位（主ボタン/副ボタン）が統一されていない。

## 4. 改修優先バックログ（本番GUI模倣を維持）

1. **文脈ヘッダ統一**
   - site/device/issue/timeを全主要画面で同形式表示。
2. **遷移理由ラベル統一**
   - Drill-downリンクに「原因→次確認先」を短文で併記。
3. **操作語彙と配置の標準化**
   - Focus/Open/Selectの用語を統一し、主アクション位置を固定。
4. **右rail密度の規格化**
   - Summary / Actions / Related の3ブロック構成に統一。

## 5. 進め方（最小リスク）

- まずAssurance/Inventory/Topologyの3画面でテンプレート化。
- 既存ルーティングと既存データ構造は維持し、文言・配置・表示規約のみを段階適用。
