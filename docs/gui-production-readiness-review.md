# GUI仕上がりレビュー（Catalyst Center 本番らしさ観点）

> 前提: 実機GUIは未確認。React/TSXとCSS定義の読解による静的レビュー。

## 完成している点

1. **App Shellの骨格は業務コンソールとして成立**
   - 左固定ナビ + 上部濃色ヘッダ + 右側Global Context Railの3面構成が明確で、情報密度の高い運用画面の基本型はできている。
   - ナビはOperate/Deliver/Integrateの業務単位でグルーピングされ、運用導線として理解しやすい。

2. **Dashboardの主従構造は最低限成立**
   - KPI行で「Open Issues」をprimaryスタイル化しており、最優先監視対象を示す意図が視覚的に出ている。
   - 「Running Discovery」を同列KPIの1つに抑えており、補助KPIとしての立ち位置は概ね妥当。

3. **Operations Hubとしての構成要素は揃っている**
   - Control bar → KPI → Primary領域 → 右railの順に情報が流れる構造になっている。
   - Task/Event、Impacted Entity、Next Actionsが横断的に繋がる設計思想は見える。

4. **Path Traceは“Drill-down”表記へ寄せる方向性がある**
   - 上部shortcutに「Path Trace (Drill-down)」を条件表示し、常時露出を避ける意図は良い。

## 未完成な点

1. **Dashboardがまだ「表を並べた管理画面」寄り**
   - PrimaryがDataTable中心で、運用ハブらしい“状況サマリ＋即応アクション”の塊に見えにくい。
   - Dashlet間の視覚差が弱く、重要度より「同じ箱の並び」に読まれやすい。
   - 本番らしさを弱める理由: Catalyst Centerの実運用感は、一覧性だけでなく優先順位の即読性が要。

2. **Control barが監視スコープ帯として弱い**
   - Time/Site/Domain/Refreshがフォーム列として並び、コンテキストを宣言する帯に見えにくい。
   - 「監視中の対象・鮮度・適用範囲」が一目で固定表示される構成になっていない。
   - 本番らしさを弱める理由: NOC画面では“今どのスコープを見ているか”の誤読防止が最重要。

3. **右側詳細railの接続感がまだ弱い**
   - Selected Task/EventとSelected Impacted Entityが独立カードとして存在し、中央選択との連動関係が視覚的に薄い。
   - どの中央テーブル選択がどのrailを更新したかの手掛かり（同調ハイライト等）が不足。
   - 本番らしさを弱める理由: 調査系UIは“選択→詳細”の因果が即時に伝わる必要がある。

4. **Global Context Railが“別レーン”に見えやすい**
   - 常時表示の右端Drawerが、ページ内右railと並列で二重化して見える可能性がある。
   - グローバル選択とページローカル選択の責務境界が、初見で直感化しづらい。
   - 本番らしさを弱める理由: railの多重化はオペレータの視線移動コストを増やす。

5. **上部shortcutsが文脈ショートカットとしては弱め**
   - Assurance/Topologyが固定リンクに近く、ページ文脈に応じた“次の一手”としての意味づけが薄い。
   - “Context shortcuts”ラベルに対して、動的性・推奨性の表現が不足。
   - 本番らしさを弱める理由: 文脈ショートカットは「なぜ今この遷移か」を示して初めて機能する。

6. **細部にプロトタイプ感が残る**
   - 記号アイコン（◻ ◎ ▤ 等）が本番風の情報記号体系に見えず、試作印象を残す。
   - 罫線・カード・均等グリッドが整いすぎて、情報優先度よりレイアウト都合が前に出る。

## 今すぐ直すべき点（優先順位順）

1. **Primary領域の再定義（最優先）**
   - Operational Focusを“最大テーブル”ではなく、優先インシデント集約+即応CTAの主役ブロックに再構成。
   - 理由: 現状は主役の強さが不足し、Operations Hubらしさを最も損ねている。

2. **Control barを“監視コンテキスト帯”へ昇格**
   - Time/Site/Domain/Refreshの入力要素より、現在スコープ・データ鮮度・更新状態の読取を優先表示。
   - 理由: 運用画面としての誤操作防止・状況共有に直結するため。

3. **中央選択と右railの連動可視化**
   - 行選択とrailカード間で見出し/強調/トレースを連動させ、選択結果の流入先を明確化。
   - 理由: 調査オペレーションの認知負荷を即時に下げられる。

4. **Path Trace導線の統一**
   - 常設リンク扱いを排し、Task/Entity選択後のdrill-down actionとしてのみ露出を統一。
   - 理由: Path Traceは文脈起点の解析導線であり、常時露出はノイズ化しやすい。

5. **Global Context Railとページ右railの役割分離を明示**
   - “Global pin”と“Page selection”のラベル/区切りを強化し、二重railの意味を明確化。
   - 理由: 同じ右側情報でも責務が違うことを初見で理解させる必要がある。

## 最優先3項目

1. Operational Focusを主役ブロックへ再構成
2. Control barを監視コンテキスト帯として再設計
3. 中央選択と右railの連動を視覚的に明示
