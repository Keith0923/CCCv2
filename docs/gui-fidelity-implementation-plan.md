# CCCv2 本番GUI再現度 改修計画

## 0. 目的

`docs/enarsi-catalyst-center-review.md` の指摘を、既存ルーティング/データ構造を維持しつつ段階的に反映する。

## 1. スコープ

### 対象画面
- Assurance
- Inventory
- Topology

### 対象外
- ルーティング構造の変更
- Seedデータ構造の変更
- 新規機能（学習専用機能を含む）の追加

## 2. 実装フェーズ

### Phase 1: 文脈ヘッダ統一（最優先）

#### 実装
- 上記3画面で `site/device/issue/time` を同形式で表示。
- クエリパラメータ優先順位を統一（画面共通ルール化）。

#### 成果物
- 共通UI部品（Context Header）
- 3画面への適用PR

#### 完了条件
- 3画面で同じ順序・同じラベルで文脈表示される。

---

### Phase 2: 遷移理由ラベル統一

#### 実装
- Drill-downリンクに「遷移理由」の短文ラベルを付与。
- ラベル文言を共通定義に集約。

#### 成果物
- 遷移理由ラベル定義
- Assurance/Inventory/Topologyの導線更新PR

#### 完了条件
- 主要導線で「なぜ次にこの画面か」が1行で読める。

---

### Phase 3: 操作語彙/配置の標準化

#### 実装
- `Focus/Open/Select` の語彙ルールを固定。
- 主アクション/副アクションの配置規約を統一。

#### 成果物
- 操作語彙ガイド
- ボタン文言・配置の調整PR

#### 完了条件
- 同種操作で別語・別配置が残っていない。

---

### Phase 4: 右rail密度の規格化

#### 実装
- 右railを `Summary / Actions / Related` の3ブロックに統一。
- 画面ごとの情報密度を上限ルールで揃える。

#### 成果物
- 右railテンプレート
- 3画面適用PR

#### 完了条件
- 右rail構造の差異が意図的なものだけになる。

## 3. リスクと対策

- リスク: 既存導線の見え方が変わり、学習者が混乱する。
  - 対策: Phaseごとに小PR化し、画面比較レビューを実施。

- リスク: 文言統一で本番GUIから離れる。
  - 対策: 本番GUI表記との差分レビューを都度実施。

## 4. 未確定事項（要確認）

1. 本番GUI準拠の優先基準
   - 配置を最優先（確定）
2. 用語統一の基準（英語優先/日本語優先）
   - 英語優先（確定）
3. 右rail情報量の許容上限（行数/ブロック数）
   - 3ブロック固定 / 各ブロック最大5行（確定）
4. Phase 1〜4の希望実施順（この順で進めてよいか）
   - 一旦OK（確定）

## 5. 次アクション

- Phase 1（文脈ヘッダ統一）は実装済み。表示差分レビューを継続。
- Phase 2（遷移理由ラベル統一）を実装し、主要導線へ適用。
- Phase 3以降は合意後に順次実施。
